# Code & Backlog Audit — 2026-07-04

Full review of the docs (PRD/ARCHITECTURE/IMPLEMENTATION_PLAN), the sidecar, the frontend, the Tauri shell, the CI workflows, and the GitHub issue backlog. Every finding below is tracked as a GitHub issue; this file is the index.

Scope: repository state at `31b05da` (post v0.2.7). Companion to the security audit in #100.

## Correctness bugs

| Issue | Finding | Severity |
|-------|---------|----------|
| [#102](https://github.com/justminime/faces-h/issues/102) | Deleting or merging a person raises an FK `IntegrityError` (→ HTTP 500) whenever any uncertain face still has `suggested_person_id` pointing at it. `merge_people`/`delete_person` never clear or re-point the column, and `PRAGMA foreign_keys=ON` is active. | **High** — breaks merge/delete for exactly the clusters users merge most |
| [#119](https://github.com/justminime/faces-h/issues/119) | The NSIS installer hooks (`customInstall`/`customUnInstall`) use **electron-builder** macro names; Tauri 2 only invokes `NSIS_HOOK_PREINSTALL`/`POSTINSTALL`/`PREUNINSTALL`. The taskkill-before-upgrade (#51/#77/#84) and Defender exclusion (#66) have silently never run. Also: no runtime supervision — if the sidecar dies mid-session the frontend keeps running as a zombie. | **High** — explains sidecar surviving upgrades and frontend-only sessions |
| [#115](https://github.com/justminime/faces-h/issues/115) | `merge_people` never rebuilds the target centroid (the code comment claiming it does is wrong). Post-merge sweep scores against the stale centroid; merged faces keep `assign_conf` computed vs. the deleted source (Rule 2 drift). | High |
| [#103](https://github.com/justminime/faces-h/issues/103) | **Rule 2 violation:** `POST /queue/{id}/confirm` promotes a face to `assigned` without recomputing `assign_conf` against the confirmed person's centroid — wrong-person confidence if the user picks someone other than the suggestion. Plus: no 404 for unknown person; inconsistent `assign_conf` NULLing on demotion. | High (reliability rule) |
| [#104](https://github.com/justminime/faces-h/issues/104) | Rescanning a photo whose mtime changed re-runs face extraction **without deleting the old face rows** → duplicate faces per rescan of any edited photo. Must be designed with #90 (crash-resume re-extraction gate). | Medium |
| [#108](https://github.com/justminime/faces-h/issues/108) | **Rule 4 gap:** the `UncertainQueue` component is never rendered — the review flow shipped in #15 is unreachable. Badge click just opens the gallery; `fetchUncertainQueue`/`confirmFace` are dead client code. Uncertain faces accumulate with no resolution path. | High (UX/reliability) |
| [#109](https://github.com/justminime/faces-h/issues/109) | Sidebar "Unnamed" count is hardcoded to `MOCK_UNNAMED_COUNT = 7` from `src/mocks/data.ts`; production domain types (`Person`/`Photo`/`FaceInfo`) also live in the mocks module. | Medium |
| [#105](https://github.com/justminime/faces-h/issues/105) | Nothing ever removes a `photos` row — photos deleted/moved on disk show as broken tiles forever, keep counting toward people, and their embeddings keep shaping centroids. Needs network-root-aware reconciliation (don't mass-mark an offline NAS as missing). | Medium |

## Performance / scalability

| Issue | Finding |
|-------|---------|
| [#106](https://github.com/justminime/faces-h/issues/106) | **The FAISS index is never used in production.** `FAISSManager` is test-only dead code; `faces.embedding_id` is never written. `assign_face` linear-scans every `people.centroid` blob per face, and every unmatched face seeds a new singleton person — assignment cost degrades toward O(n²). Contradicts ARCHITECTURE D-07/OD-06 and the 5TB/1M-embedding target (#19 benchmarked FAISS in isolation, not the real path). |
| [#114](https://github.com/justminime/faces-h/issues/114) | No disk cache for thumbnails/face crops — every request decodes the full-resolution original (incl. over SMB for network roots). Cache under `{data_dir}/cache/` with mtime validation + LRU cap. |
| [#110](https://github.com/justminime/faces-h/issues/110) | Every `scan_progress` WebSocket event bumps `scanVersion`, which refetches the full people list + queue count — ~1 refetch per 10 scanned photos, i.e. ~10k refetches on a 100k-photo scan, competing with the scanner for SQLite. Needs throttling. |
| [#113](https://github.com/justminime/faces-h/issues/113) | Grouped sidecar robustness: `/scan/start` double-start race; module-global `ScanStatus` stomped across roots; 3+ commits per face during extraction; schema DDL + exception-swallowing migrations re-executed on every request; ETA math ignores skipped files. |

## Missing features vs. architecture

| Issue | Finding |
|-------|---------|
| [#107](https://github.com/justminime/faces-h/issues/107) | `config.json` (model selection + thresholds) is not implemented. Thresholds 0.68/0.50 are duplicated in `clustering.py`, `reeval.py`, and imported privately by `transfer.py`; `ml/factory.py` is dead code, so the "swappable ML backend" is not actually swappable. OD-02 ("tune after first real-world scan") is impossible without this. |
| [#111](https://github.com/justminime/faces-h/issues/111) | OD-04's resolved decision — skip faces below ~20px and log a skipped count — is unimplemented; every detection is stored and clustered regardless of size or `det_score`, feeding junk into clusters and the uncertain queue. |
| [#118](https://github.com/justminime/faces-h/issues/118) | Startup UX: no app-level "what am I doing and why" status during the (normal) 30–180 s engine start / model download / retry window — users only see "Sidecar disconnected — reconnecting…" in the activity log. Root of the #101-style "app looks broken" reports. |
